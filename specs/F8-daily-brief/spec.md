# AI Daily Brief

**ID:** F8-daily-brief
**Status:** Build (spec & plan approved 2026-07-02)
**Owner:** Marco

## Intent

The app classifies and summarizes articles individually, but nothing answers
the reader's real question: "what actually matters today?" Every card carries
equal visual weight, so getting the day's picture still means scanning dozens
of cards. Add an editorial brief — 3–5 synthesized bullets generated by Claude
at ingest time from that run's new articles, each bullet citing its source
articles — rendered at the top of the page. This is also the feature that best
showcases the project's AI engineering: it moves beyond per-item classification
to cross-article synthesis with citations, while staying fully static and
adding only cents to the ingest cost.

## User stories

- As a daily reader, I want a short synthesized brief of the day's most
  significant developments, so that I get the headline picture in 20 seconds
  before deciding what to read in depth.
- As a skeptical reader, I want each brief bullet to link to the articles it
  draws from, so that I can verify claims and dig deeper.
- As the maintainer, I want the brief generated inside the existing ingest run,
  so that the site stays a static build with no server runtime and negligible
  added cost.

## Acceptance criteria

- WHEN an ingest run classifies at least 3 new articles THEN the pipeline
  SHALL generate a brief of 3–5 bullets (≤ 40 words each) synthesized only from
  that run's newly classified articles, and SHALL store it in `news.json` with
  its own generation timestamp. *(Amended during Build: runs with 1–2 new
  articles cannot yield a valid 3-bullet brief, so they skip the call and
  carry the previous brief forward — observed live before the amendment.)*
- WHEN a brief bullet draws on specific articles THEN it SHALL carry references
  (article URLs or indices) and the UI SHALL render those references as links
  to the original articles.
- WHEN an ingest run finds no new articles THEN the previous brief SHALL be
  retained unchanged and the UI SHALL show its original date.
- WHEN the All view renders and a brief exists THEN the brief SHALL appear
  above the article sections, styled within the existing dark-academic design
  language (see `.interface-design/system.md`).
- WHEN brief generation fails (API error, malformed response) THEN the ingest
  run SHALL log the failure and complete successfully without a brief — never
  failing the run or corrupting existing data.
- WHEN the brief is generated THEN important/notable articles SHALL be weighted
  toward inclusion, but the prompt SHALL allow the model to surface an
  unmarked article if it is clearly the day's biggest story.

## Non-goals

- No per-category briefs — one global brief only.
- No browsable archive of past briefs (only the latest is rendered; whether
  older ones are retained in JSON is an open question).
- No email/notification delivery of the brief (that is F5's territory).
- No streaming or client-side generation — the client only reads static JSON.

## Open questions

Resolved at Gate 1 (2026-07-02) with recommended defaults — override any of
these at Gate 2 (plan review) if you disagree.

- [x] Model choice — **`claude-opus-4-8`** for the single synthesis call
      (current recommended default model; best synthesis quality; roughly
      $0.10–0.25 per run at expected input sizes). Haiku stays for bulk
      classification; dropping the brief call to Haiku remains a one-line
      change if cost ever matters.
- [x] Brief history — **store only the latest brief** in `news.json` (YAGNI;
      a rolling archive becomes natural after the roadmapped SQLite
      migration).
- [x] Placement — **All view only**, above the article sections.
